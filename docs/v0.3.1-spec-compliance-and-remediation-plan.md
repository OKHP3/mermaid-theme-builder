## 3. Where the Implementation Exceeds the Notion Proposal

1. **Capability metadata in Examples tab.** Each example preview now shows support status badge, theme confidence, notation compliance, and (when present) the honest BPMN-style warning. The proposal only required these in the Reference tab.
2. **Reference tab "Further reading" footer.** Direct links to the Visual Language Diagram Taxonomy in Notion and the OKHP3 GitHub repo. Not in the proposal.
3. **Brand-example auto-load on palette switch** (from follow-up #18). Selecting OverKill Hill / Glee-fully / AskJamie loads the matching brand flowchart when the textarea is empty or unedited.
4. **Assessment document** (`docs/ux-restructure-assessment.md`). Captures maturity, plan alignment, deficiencies, planned correction, non-goals.
5. **Governance files** are present (`LICENSE`, `AGENTS.md`, updated `README.md`). The proposal scored these 1/5; they are now in good shape, lifting that dimension to ~3/5.

## 4. Identified Deficits (with remediation)

### Deficit #1 — Apply tab warning is over-noisy
**Symptom.** A first-time user pasting a healthy `flowchart TD` sees an amber warning banner reading *"Best-supported family for rich theme styling, classDef, subgraphs, links, and visual governance."* That string is the capability's positive description, not a warning. The proposal says "Render warnings (only if present)."

**Root cause.** `ApplyTab.tsx` line 113–115 pushes `detection.capability.warning` into the warnings array unconditionally whenever the diagram family is recognized. Several capability records use the `warning` field for descriptive notes about *strengths*, not problems.

**Fix.**
- Only surface `detection.capability.warning` when the diagram is non-native: i.e. `supportStatus !== "native"` OR `themeConfidence` is `"generic-only" | "low" | "not-applicable"` OR `stability !== "stable"`.
- Continue to surface true render-safety warnings (existing init directives, non-printable chars, long labels) regardless.
- Effort: ~10 minutes. File: `src/pages/tabs/ApplyTab.tsx`. No data changes required.

### Deficit #2 — DiagramInventory not surfaced in Examples tab
**Symptom.** The Notion proposal explicitly says *"DiagramInventory → Examples + Reference tabs"*. Currently the inventory only lives in Reference. Examples shows only loadable examples, not the full inventory of supported diagram families.

**Fix.**
- Add a small "Browse all supported families" link or collapsible at the top of the Examples tab list that opens DiagramInventory in an embedded drawer, OR add a secondary view toggle inside Examples (Library / Inventory).
- Reuse the existing `<DiagramInventory embedded />` — no logic changes.
- Effort: ~30 minutes. File: `src/pages/tabs/ExamplesTab.tsx`.

### Deficit #3 — Mobile Apply textarea defaults too tall
**Symptom.** `min-h-[140px]` renders roughly 10 visible lines on a typical phone viewport. Proposal says "code input collapses to 4 lines with expand."

**Fix.**
- Set the mobile minimum to roughly 4 visible lines (`min-h-[88px]` at the current font size) and add an expand toggle button that grows the textarea to `min-h-[60vh]` on tap.
- Keep the desktop layout (`md:min-h-0`) unchanged.
- Effort: ~20 minutes. File: `src/pages/tabs/ApplyTab.tsx`. Add a small `useState<boolean>` for expand.

### Deficit #4 — Preview pinch-to-zoom not implemented
**Symptom.** The proposal says "Preview gets pinch-to-zoom" on mobile. The current `MermaidPreview` renders a static SVG with no touch gesture handling.

**Fix.**
- Wrap the preview SVG container in a transform wrapper that responds to touch pinch gestures (`react-zoom-pan-pinch` is the lightest viable dep at ~6 KB gzipped) OR use a CSS `touch-action: pinch-zoom` container with native browser zoom on the SVG.
- Native CSS approach is preferred — zero new dependencies, preserves the static/no-backend constraint cleanly.
- Acceptance: on mobile, two-finger pinch on the preview pane scales the diagram between 0.5× and 4× without breaking Mermaid rendering.
- Effort: ~45 minutes (CSS-only) or ~90 minutes (library-based). File: `src/components/MermaidPreview.tsx` plus a thin wrapper.

## 5. Out-of-Scope Items Worth Logging (not part of this remediation)

- **Export system → 3/5.** Notion baseline noted "no file download". Adding `.svg` / `.png` / `.md` file downloads would lift this to 4/5 but is a separate feature, not a UX-restructure deficit.
- **Diagram detection → 3/5.** Family-specific themeVariables would lift this to 4/5. Belongs to a future logic-layer release, not V0.3.1.
- **Pinch-to-zoom preservation across re-renders.** If pinch-zoom is added, ensure Mermaid re-renders (on theme change) don't reset the zoom transform.
