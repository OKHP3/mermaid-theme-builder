# Mermaid Style Builder — Product Requirements Document

**Product name:** Mermaid Style Builder
**Tagline:** Define your brand. Style your diagrams. Stop fighting with LLM outputs.
**Version:** v0.1 (Alpha)
**Author:** Jamie Hill, OverKill Hill P3
**Date:** April 2026
**Repo:** github.com/OKHP3/mermaid-style-builder
**Deployment:** overkillhill.com/projects/mermaid-style-builder/

---

## 1. Problem Statement

When LLMs generate Mermaid diagrams, the visual output is effectively random. Every diagram gets a different color palette — Mermaid defaults, hallucinated hex codes, or no styling at all. Two diagrams generated in the same conversation look like they were made by different tools.

Users attempt to fix this through follow-up prompts ("make it blue," "use my brand colors"), but this creates a costly, unreliable cycle: each prompt burns tokens, the LLM may partially comply while inventing other colors, and structural content sometimes breaks during restyling. Maintaining consistency across a multi-diagram deliverable through follow-up prompts is essentially impossible.

The root cause: Mermaid has powerful theming capabilities (themeVariables, classDef, frontmatter config) but no user-facing workflow for defining, applying, and reusing a visual identity across diagrams and across LLM conversations.

## 2. Target Users

**Primary:** Enterprise architects, technical writers, and consultants who generate Mermaid diagrams through AI assistants (Claude, ChatGPT, Copilot, Gemini, Perplexity) and need consistent, professional visual output across multi-diagram deliverables.

**Secondary:** Developers using Mermaid in markdown documentation who want branded diagram output without learning Mermaid's theming syntax.

**Tertiary:** The Mermaid.ai community — practitioners who work with Mermaid directly and want a faster way to create and manage custom themes.

## 3. Solution Overview

A client-side web application hosted on GitHub Pages that provides three operational modes:

### Mode 1: Theme Composer (pre-prompt scaffolding)

The user defines brand colors through interactive color pickers. The tool generates a complete, portable Mermaid style specification — YAML frontmatter, init directive, classDef library, subgraph style tiers — that the user downloads as a markdown file or copies to clipboard. This file is designed to be uploaded as context to any LLM or pasted into a prompt, so the LLM generates on-brand diagrams on the first response.

**Analogy:** Starting a PowerPoint from a branded template.

### Mode 2: Theme Applicator (post-generation styling)

The user pastes an existing Mermaid diagram (plain, unstyled, or generated by an LLM). The tool detects the diagram type, injects the active theme configuration, and renders a live preview. The user copies the styled code.

**Analogy:** Applying a Word style template to an existing document.

### Mode 3: Theme Extractor (reverse engineering + two-way binding)

The user pastes a diagram that already has styling (classDefs, init directives, inline styles). The tool extracts the color values, maps them to design tokens, and populates the color picker panel. Adjustments to pickers update the diagram code in real time. Loading a new diagram updates the pickers. The binding is bidirectional.

**Analogy:** Inspecting a PowerPoint's color theme and modifying it.

## 4. What This Product Is Not

- Not a diagram editor. Content (nodes, edges, labels) passes through unchanged.
- Not a rendering service. Preview is a validation convenience; primary output is styled text.
- Not an LLM. Does not generate diagrams from natural language.
- Not a competitor to Mermaid.ai. Generates configuration, does not host or manage diagrams.

## 7. Technical Architecture

### Stack

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Build tool | Vite | Lightweight, fast, produces clean static output for GitHub Pages |
| Language | Vanilla JavaScript (ES modules) | No framework dependency. Keeps bundle small. GitHub Pages compatible. |
| Styling | CSS (no preprocessor) | Minimal styling needs. CSS custom properties for theming the tool itself. |
| Diagram rendering | mermaid.js via CDN | MIT licensed. Same library used by GitHub, VS Code, Loop, Notion. |
| Hosting | GitHub Pages | Free. Static. Matches existing overkillhill.com infrastructure. |

### File Structure

```
mermaid-style-builder/
  README.md
  LICENSE
  package.json
  vite.config.js
  index.html
  src/
    main.js              # Entry point, app initialization
    tokens.js            # Color token state management
    presets.js            # Built-in palette presets
    generator.js          # Output generation (YAML, init, classDef, subgraph)
    injector.js           # Diagram type detection and theme injection
    extractor.js          # Theme extraction from existing diagrams
    preview.js            # Mermaid.js preview rendering
    ui.js                 # DOM manipulation, event binding, tabs
    utils.js              # Color math (luminance, darken, contrast)
  styles/

  assets/
    og-image.png          # Open Graph image for social sharing
  dist/                   # Built output (generated by Vite, not committed)
  docs/
    PRD.md                # This document

    DESIGN_SYSTEM.md      # BFS Mermaid Design System v1.0
    BUILD_SPEC.md         # Replit-consumable build specification
```

### Dependency on mermaid.js

Mermaid.js is consumed via CDN in the production build, not as an npm dependency. The `index.html` includes:

```html
<script src="https://cdn.jsdelivr.net/npm/mermaid@11.14.0/dist/mermaid.min.js"></script>
```

For the Vite dev server, mermaid is installed as a dev dependency (`npm install -D mermaid`) so the dev environment works locally, but the production build uses the CDN version. This avoids bundling mermaid into the output and keeps the static deployment lightweight.

### Data Flow

```
Color Pickers (user input)
    |
    v
Token State (tokens.js)
    |
    +---> Generator (generator.js) ---> Output Tabs (YAML, init, classDef, subgraph, full MD)
    |
    +---> Injector (injector.js) <--- Diagram Textarea (user paste)
    |         |
    |         v
    |    Styled Code ---> Copy to Clipboard
    |         |
    |         v
    |    Preview Panel (preview.js) ---> mermaid.render() ---> SVG output
    |
    +---> Extractor (extractor.js) <--- Diagram Textarea (existing styled diagram)
              |
              v
         Token State (populates pickers from extracted values)
```

## 8. UI Layout

### Primary Layout (Desktop)

Two-column layout.

**Left column (40% width):** Color token inputs organized into sections (Brand Blues, Fills and Surfaces, Chrome and Signals, Typography). Preset buttons at top. Below the inputs, a tab bar for output selection (YAML, Init, classDef, Subgraph, Full System). Below that, the selected output code block with copy button.

**Right column (60% width):** At top, a mode selector (Compose / Apply / Extract). Below that, the diagram paste textarea (visible in Apply and Extract modes, hidden in Compose mode). Below that, the live preview panel showing either: node class previews (Compose mode), styled diagram preview (Apply mode), or styled diagram with bidirectional updates (Extract mode).

### Mode Behavior

| Element | Compose Mode | Apply Mode | Extract Mode |
|---------|-------------|------------|--------------|
| Color pickers | Active, user-editable | Active, user-editable | Active, auto-populated then editable |
| Diagram textarea | Hidden | Visible, accepts paste | Visible, accepts paste |
| Output tabs | Visible, all tabs | Visible, "Styled Code" tab added | Visible, all tabs |
| Preview panel | Node class samples + subgraph tier samples | Live diagram render | Live diagram render with bidirectional updates |

## 9. User Workflows

### Workflow A: "I want to create a brand theme from scratch"

1. User opens the tool (default: Compose mode)
2. User selects a preset or adjusts color pickers manually
3. Node class preview updates in real time
4. User selects output tab (YAML, Init, classDef, etc.)
5. User clicks Copy to get the configuration block
6. User pastes into Claude/ChatGPT/Copilot as context for future diagram generation
7. (Optional) User clicks Download to save the full design system as a .md file

### Workflow B: "I have a bland diagram and want to style it"

1. User switches to Apply mode
2. User pastes Mermaid code into the textarea
3. Tool detects diagram type, injects active theme
4. Preview panel renders the styled diagram
5. User adjusts colors if needed (preview updates live)
6. User copies the styled code from the output panel
7. User pastes styled code into their target destination

### Workflow C: "I have a styled diagram and want to extract/modify the theme"

1. User switches to Extract mode
2. User pastes a diagram containing existing styling
3. Tool parses the styling and populates color pickers
4. User adjusts colors (preview and code update bidirectionally)
5. User copies the refined styled code or exports the theme
