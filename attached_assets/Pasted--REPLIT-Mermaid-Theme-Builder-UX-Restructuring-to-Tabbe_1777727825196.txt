## CURRENT CONTEXT

Mermaid Theme Builder has reached a functional public-alpha state. It is live on GitHub Pages, has brand presets, a color editor, live Mermaid preview, diagram detection, capability registry, prompt scaffold modal, example loading, and documentation.

However, the current UX has reached the limit of the single-screen layout. The app is functionally promising but visually and cognitively overloaded.

The current single-screen layout puts too much on one page:

- palette selector
- full color editor
- font controls
- code input
- live preview
- original/themed output
- export buttons
- prompt scaffold modal
- diagram inventory
- capability notes
- class browser
- example selector
- warnings
- metadata controls
- documentation-adjacent reference content

This barely works on a wide desktop browser and fails on mobile. The mobile experience is not simply “a little cramped”; it is structurally overloaded. The desktop experience is also doing too much in one window.

The next goal is not to add features. The next goal is to restructure the application into a workflow-oriented tabbed architecture.

## PROBLEM TO SOLVE

The current UI is trying to solve all jobs at once.

The product’s core jobs are distinct:

1. Apply a theme to existing Mermaid code.
2. Compose a reusable theme/bootstrap without needing a diagram.
3. Browse examples and load one into the editor.
4. Understand Mermaid capability, support status, and limitations.

Those are four different user modes. They should become four tabs.

## TARGET UX ARCHITECTURE

Replace the current everything-window with a four-tab interface:

1. APPLY
2. COMPOSE
3. EXAMPLES
4. REFERENCE

Use a simple tab state for V0.3.1. Do not introduce a complex router unless the current app already has routing and it is cleaner to use it.

Suggested state shape:

type AppTab = "apply" | "compose" | "examples" | "reference";

Default active tab:

apply

All four tabs must share the same app-level state:
- selected palette
- custom colors
- font family
- theme name
- watermark/metadata settings
- current Mermaid input code
- detected diagram family
- generated styled Mermaid
- warnings
- selected example

The restructuring is primarily layout and information architecture. Existing logic should be preserved and relocated.

## TAB 1: APPLY

Apply is the default tab and must load first.

This tab is the core workflow:

paste Mermaid → choose theme → preview → copy/export

Apply must be the simplest screen in the app.

### Apply tab purpose

Answer one question:

“What happens when I paste Mermaid and apply a theme?”

### Apply desktop layout

Use a two-pane layout.

Top:

[Tab Bar: Apply* | Compose | Examples | Reference]

Main body:

Left pane:
- Compact horizontal palette strip
- Mermaid code input textarea

Right pane:
- Original / Themed toggle
- Live Mermaid preview

Bottom action bar:
- Edit Colors
- Styled Code
- Markdown
- Prompt Scaffold

Warnings:
- Display inline above the bottom action bar when relevant.
- Warnings should be concise.
- Full explanatory capability/reference content belongs in Reference, not Apply.

### Apply desktop visual structure

Use this as the intended layout:

Use this structure:

┌───────────────────────┐
│ Palette strip          │
│ horizontal scroll      │
├───────────────────────┤
│ Code input             │
│ 4 visible lines        │
│ expandable             │
├───────────────────────┤
│ Live Preview           │

└───────────────────────┘

Mobile rules:
- No side-by-side layout.
- No visible full color editor by default.
- No capability table in Apply.
- No ClassBrowser in Apply.
- No DiagramInventory in Apply.
- No large example browser in Apply.
- No dense export panel in Apply.
- Action buttons must be thumb-friendly.
- Preview should not be squeezed to an unusable height.

### What moves out of Apply

Move these out of the Apply default view:

- Full color editor with all color fields
- Font family editor
- ClassBrowser
- DiagramInventory
- CapabilityNote full explanation
- Example selector/gallery
- Large prompt scaffold UI
- Long documentation/reference content

Apply may still show:
- short detected family badge
- short support/confidence badge
- concise warning
- action buttons that open modals/panels

### Apply color editing

The full color editor should not be visible in Apply.

The Edit Colors button should open:
- slide-out panel on desktop, or
- full-screen modal/drawer on mobile

This edit panel should allow quick adjustment without permanently occupying the Apply tab.

### Compose mobile behavior

- Color editor can take the full screen.
- Preview should appear below the editor, not squeezed beside it.
- Export buttons should be fixed or sticky at bottom.
- Preset selection can be a compact dropdown or horizontally scrollable cards.
- Avoid dense multi-column color editor on mobile.

### Compose constraints

This tab is allowed to show the full color editor because that is its job.

This tab should not require pasted Mermaid code.

### Examples behavior

When the user selects an example:
- Show preview using the currently selected theme.
- Show support status.
- Show warnings if beta, renderer-dependent, emulated, or unsupported.
- Show whether notation is Mermaid-native or approximation only.

When the user clicks “Load into Apply Tab”:
- Set the current input code to the example’s Mermaid code.
- Switch active tab to Apply.
- Preserve selected palette and custom colors.
- Trigger detection and preview update.

When the user clicks “Copy Raw Code”:
- Copy the unstyled or baseline example code.

### Examples mobile behavior

- Use card list.
- Preview can appear below selected card.
- “Load into Apply” should be prominent.
- Avoid dense side-by-side layout.

### Reference constraints

Reference must not clutter Apply.

Reference should contain explanatory content that is currently crowding the main screen.

Capability warnings should be honest:
- Native means Mermaid supports it directly.
- Partial means limited/variant support.
- Emulatable means visually approximate only.
- Gap means not meaningfully supported.
- External means better handled by specialized tooling.

BPMN warning must remain:

“Mermaid flowcharts can approximate BPMN-like visual flows, but they do not implement BPMN 2.0 semantics.”

## IMPLEMENTATION APPROACH

### Step 1: Create tab navigation

Add a top-level tab bar component.

Use React state for active tab unless existing app architecture makes a lightweight route cleaner.

Suggested type:

type AppTab = "apply" | "compose" | "examples" | "reference";

const [activeTab, setActiveTab] = useState<AppTab>("apply");

The default active tab must be Apply.

Desktop:
- Top tab bar.

Mobile:
- Top tab bar is acceptable if usable.
- Prefer bottom navigation if simple to implement and it improves usability.
- Do not create horizontal overflow.

### Step 2: Extract existing UI into tab-specific components

Create or refactor into tab-specific components:

- ApplyTab
- ComposeTab
- ExamplesTab
- ReferenceTab

The exact file names can follow the repo’s conventions.

Suggested responsibilities:

ApplyTab:
- compact palette strip
- Mermaid input
- preview
- short warning summary
- export action buttons
- Edit Colors slide-out/modal trigger

ComposeTab:
- full palette selector
- full color editor
- font controls
- theme name / metadata
- generic preview
- bootstrap/prompt scaffold exports

ExamplesTab:
- example list
- example preview
- support badges
- Load into Apply action
- Copy Raw Code action

ReferenceTab:
- DiagramInventory
- ClassBrowser
- CapabilityNote
- capability registry search/filter
- support/limitation explanations
- Notion/GitHub links

### Step 3: Lift shared state

Ensure these are shared across tabs:

- selectedPaletteId
- customColors
- inputCode
- detectedFamily
- generatedOutput
- selectedExample
- fontFamily
- themeName
- watermarkEnabled
- metadataCommentsEnabled

Do not duplicate state in each tab in a way that causes divergence.

When the user changes palette in Compose, Apply should use that palette.

When the user loads an example in Examples, Apply should receive that code.

When the user edits colors from Apply’s modal, Compose should reflect those custom colors.

### Step 4: Preserve existing logic

Do not rewrite business logic unless required by the refactor.

Do not rewrite:
- themeEngine
- detector
- palettes
- MermaidPreview
- ColorSwatch

Relocate and reuse.

Any bug fixes should be minimal and documented.

### Step 5: Mobile-specific adjustments

Implement responsive behavior intentionally.

Mobile requirements:

- No horizontal scrolling.
- No desktop side-by-side panes.
- Apply tab uses a vertical layout.
- Code input starts compact, approximately 4 visible lines, with expand behavior if feasible.
- Color editor opens as modal/drawer/full-screen panel.
- Export actions appear in a compact fixed or sticky action bar.
- Palette strip is horizontally scrollable compact swatches.
- Reference inventory uses cards, not dense table.
- Examples use cards/list, not two-column table.
- Preview should have enough height to be useful.
- Pinch/zoom is desirable if feasible, but do not destabilize Mermaid rendering to implement it.

### Step 6: Compact palette strip in Apply

The Apply tab palette selector must be compact.

Use:
- small swatches/circles/buttons
- selected highlight
- tooltip or short label if available
- horizontal scroll on mobile

Do not use the full preset card grid in Apply.

The full card grid belongs in Compose.

### Step 7: Export behavior

Apply should have quick export actions:

- Styled Code
- Markdown
- Prompt Scaffold

These can open existing modal/panel behavior.

Compose should export:
- Bootstrap Markdown
- Prompt Scaffold

Export surfaces must preserve current functionality.

### Step 8: Inventory and examples

Do not remove registry or inventory work.

Move it into the Reference and Examples tabs.

The capability registry should remain visible and useful, but not in the Apply default workflow.

Examples should become more productized:
- selectable
- previewable
- loadable into Apply

If some examples are metadata-only or pending, label them honestly:
- Pending
- Renderer-dependent
- Beta
- Approximation only

Do not claim support for unsupported formal notations.

## SUCCESS CRITERIA

The refactor is successful when:

- Apply tab shows only the core paste/theme/preview/export workflow.
- A first-time user can paste Mermaid and copy styled output in under 30 seconds.
- Color editor is not visible by default in Apply.
- ClassBrowser is not visible in Apply.
- DiagramInventory is not visible in Apply.
- CapabilityNote full explanation is not visible in Apply.
- Examples are not crowding Apply.
- Compose tab can generate a bootstrap/scaffold without pasted diagram code.
- Examples tab can preview and load examples into Apply.
- Reference tab contains DiagramInventory, ClassBrowser, CapabilityNote, and support/limitation guidance.
- All four tabs share palette and code state.
- Mobile Apply tab is usable without horizontal scrolling.
- Mobile color editor opens as modal/drawer/full-screen panel.
- Desktop feels calmer and workflow-driven.
- Existing tests/build pass.
- GitHub Pages deploy remains functional.
- No employer references are introduced.
- No third-party corporate theme presets are introduced.
