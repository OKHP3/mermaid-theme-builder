## PHASE 1 — WORKFLOW-BASED TABBED UX ARCHITECTURE
Replace the overloaded single-screen interface with four primary tabs:
1. APPLY
2. COMPOSE
3. EXAMPLES
4. REFERENCE
Default tab:
APPLY
Use React state unless an existing routing pattern makes route-based tabs cleaner.
Suggested state:
type AppTab = "apply" | "compose" | "examples" | "reference";
const [activeTab, setActiveTab] = useState<AppTab>("apply");
All tabs must share app state:
- selected palette
- custom colors
- font family
- theme name
- metadata comments enabled
- watermark/attribution enabled
- current Mermaid input code
- detected diagram family
- generated styled code
- selected example
- warnings
Do not duplicate state in a way that causes divergence.
### Tab 1: APPLY
Purpose:
Paste Mermaid, apply a theme, preview, copy/export.
This is the default and simplest workflow.
Desktop layout:
- compact horizontal palette strip
- code input textarea
- live preview
- original/themed toggle if already supported
- short warnings
- bottom action bar:
  - Edit Colors
  - Styled Code
  - Markdown
  - Prompt Scaffold
Move OUT of Apply:
- full color editor
- full palette card grid
- font editor
- ClassBrowser
- DiagramInventory
- full CapabilityNote explanations
- large example selector/gallery
- documentation/reference content
Apply should answer:
“What happens when I paste Mermaid and apply a theme?”
Color editor in Apply:
- opened by Edit Colors button
- slide-out panel on desktop
- full-screen modal/drawer on mobile
Mobile Apply:
- no horizontal scrolling
- compact palette strip
- code input initially compact, about 4 visible lines, expandable if feasible
- preview takes meaningful vertical space
- bottom action bar fixed or sticky
- no dense tables
- no side-by-side layout
