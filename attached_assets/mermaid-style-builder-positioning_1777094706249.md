# Mermaid Diagram Styling — Problem, Solution, and Product Positioning

## The Problem

When an LLM generates a Mermaid diagram in response to a prompt, the visual output is effectively random. The model focuses on structural correctness — the right nodes, the right edges, the right relationships — but treats styling as an afterthought. Every diagram gets a different color palette. Some get Mermaid's default pastels. Some get hallucinated hex codes the model invents on the fly. Some get no styling at all.

The result is that two diagrams generated five minutes apart in the same conversation look like they were made by different people using different tools.

This matters because diagrams are communication artifacts. They go into deliverables, presentations, documentation packages, and architecture reviews. When a package contains five diagrams and each one uses a different visual language, the package looks unfinished. The inconsistency undermines the credibility of the content.

### The follow-up prompt trap

The instinctive response is to prompt the LLM again: "Make it blue. Use my brand colors. Match the other diagram." This creates a costly and unreliable cycle:

- Each follow-up prompt burns tokens and time
- The LLM may partially comply, changing some colors but inventing others
- Structural content sometimes breaks during restyling attempts
- The user ends up managing two concerns simultaneously — content accuracy and visual consistency — when these should be independent layers
- Across multiple diagrams in a package, maintaining consistency through follow-up prompts is essentially impossible without exact copy-paste of prior instructions

The fundamental issue is that **Mermaid diagrams lack a portable, reusable visual identity system.** PowerPoint has templates. Word has styles. CSS has design tokens. Mermaid has themeVariables and classDef — powerful mechanisms — but no user-facing workflow for defining, applying, and reusing a visual standard across diagrams and across LLM conversations.

### Who experiences this problem

- Enterprise architects generating architecture diagrams through AI assistants
- Technical writers producing documentation packages with multiple diagrams
- Consultants building deliverables where visual consistency signals professionalism
- Developers using Mermaid in markdown documentation who want branded output
- Anyone using Mermaid.ai, Claude, ChatGPT, Copilot, Gemini, or Perplexity to generate diagrams and wanting the output to look intentional rather than accidental

## The Solution

A web-based tool and companion style specification system that gives users control over Mermaid diagram styling at three points in the diagram lifecycle:

### Mode 1: Pre-prompt scaffolding (before the diagram exists)

The user defines their visual identity — brand colors, font, stroke weights, semantic class assignments — through an interactive color picker interface. The tool generates a portable markdown file containing the complete Mermaid style specification: YAML frontmatter, init directive, classDef library, subgraph style tiers, and AI instruction rules.

This file is designed to be uploaded as context to any LLM (Claude Projects, Custom GPTs, Copilot agents, Mermaid.ai project files) or pasted directly into a prompt. When the LLM generates a Mermaid diagram with this context loaded, the output arrives pre-styled on the first response. No follow-up prompts needed. The visual identity is locked before the content conversation even begins.

This is the "template" workflow — analogous to starting a PowerPoint from a branded template rather than a blank deck.

### Mode 2: Post-generation styling (after the diagram exists)

The user pastes an existing Mermaid diagram — one they wrote themselves, one generated by an LLM, or one copied from documentation — into the tool. The tool detects the diagram type, injects the theme configuration (frontmatter or init directive) and applicable classDef declarations, and renders a live preview of the styled result. The user copies the styled code to paste wherever they need it.

This is the "apply theme" workflow — analogous to applying a Word style template to an existing document.

### Mode 3: Theme extraction and two-way adjustment (from an existing styled diagram)

The user pastes a diagram that already has styling — classDefs, inline styles, init directives — and the tool extracts the color values, maps them to the design token structure, and populates the color picker panel. The user can then see the visual identity as an editable palette, adjust individual tokens, and watch the diagram preview update in real time. Changes to the color pickers update the diagram code. Loading a new diagram updates the color pickers. The binding is bidirectional.

This is the "extract and refine" workflow — analogous to inspecting an existing PowerPoint's color theme and modifying it.

## What the tool is not

- It is not a diagram editor. It does not create, modify, or manage diagram content (nodes, edges, labels, relationships). The user's content passes through unchanged.
- It is not a rendering service. The preview is a convenience for validation. The primary output is styled text — Mermaid code with theme configuration injected — that the user takes elsewhere to render.
- It is not an LLM. It does not generate diagrams from natural language. It styles diagrams that already exist or prepares styling context for LLMs that will generate diagrams.

## Value proposition

**For individual practitioners:** Stop fighting with LLM outputs to get consistent diagram styling. Define your visual identity once, reuse it everywhere, across every AI tool and every conversation.

**For teams and enterprises:** Establish a diagram visual standard that any team member can apply without understanding Mermaid's theming syntax. The tool abstracts the complexity of themeVariables, classDef, and frontmatter configuration behind a color picker interface.

**For the Mermaid ecosystem:** This is the missing user-facing workflow layer. Mermaid's theming engine is powerful but requires manual authorship of configuration code. This tool bridges the gap between "Mermaid supports custom themes" and "I can actually create and manage one."

## Naming considerations

The name should communicate three things:

1. It is for Mermaid diagrams specifically
2. It controls visual appearance / styling / branding
3. It produces something reusable — a system, not a one-off tweak

### What the tool does, in plain language

"Define your brand colors, generate a complete Mermaid style system, apply it to diagrams, and keep everything visually consistent."

### Name candidates and tradeoffs

| Name | Strengths | Weaknesses |
|------|-----------|------------|
| Mermaid Style Builder | Clear, searchable, describes the action | "Style" is slightly generic |
| Mermaid Brand Builder | Emphasizes the brand consistency angle | Might imply logo/identity work beyond diagrams |
| Mermaid Theme Builder | Directly maps to Mermaid's own terminology (theme, themeVariables) | "Theme" undersells — this builds more than a theme |
| Mermaid Styler | Short, clean, verb-like | Could be confused with a CSS tool |
| Mermaid Diagram Styler | Explicit about what gets styled | Longer, slightly clunky |
| Mermaid Style Forge | Memorable, implies craftsmanship | "Forge" may confuse non-native English speakers |
| Mermaid Style System | Emphasizes the system/reusability aspect | Sounds like documentation, not a tool |
| Mermaid Palette | Simple, visual, intuitive | Undersells — it produces more than a color palette |
| Mermaid Brand Kit | Emphasizes the deliverable | "Kit" implies a download, not an interactive tool |
| MermaidCSS | Draws the CSS/design-system analogy directly | Technically misleading — it doesn't produce CSS |

### SEO and discoverability context

Users searching for this solution would likely use queries like:
- "mermaid diagram custom colors"
- "mermaid theme generator"
- "mermaid brand colors tool"
- "mermaid styling tool"
- "custom mermaid theme builder"
- "mermaid diagram branding"

The words "mermaid," "theme" or "style," and "builder" or "generator" appear across most of these queries. A name containing at least two of those three terms would have the strongest organic discoverability.

### Recommendation

**Mermaid Style Builder** or **Mermaid Theme Builder** are the strongest candidates. Both are immediately understood, search-friendly, and accurately describe what the tool does.

"Theme Builder" has the advantage of using Mermaid's own vocabulary — "theme" is what Mermaid calls its visual configuration system. Someone reading Mermaid's docs about `themeVariables` and searching for a tool to help them build one would find "Theme Builder" intuitive.

"Style Builder" has the advantage of being broader — it implies the tool handles more than just the theme block (which it does — it also generates classDefs, subgraph tiers, AI instructions, and the full design system document).

Either works. The subtitle or tagline can carry the nuance the name leaves out:

> **Mermaid Style Builder**
> Define your brand. Style your diagrams. Stop fighting with LLM outputs.

or

> **Mermaid Theme Builder**
> A visual identity system for Mermaid diagrams. Build once, apply everywhere.
