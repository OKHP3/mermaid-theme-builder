## SECTION 1: NEO LOOK SELECTOR (Critical Priority)

### Problem
Mermaid v11.14.0 introduced a `look` config option with three rendering styles: `default` (classic flat), `neo` (modern with drop shadows and enhanced borders), and `handDrawn` (sketch/rough style). The Theme Builder does not expose this at all. Every palette currently generates only `default` look. This means users see only 7 visual outputs when 21 are available (7 palettes x 3 looks).

### Implementation

#### 1.1 Add `look` to shared state

In the app state (context, store, or wherever palette state lives), add:

```tsx
type DiagramLook = 'default' | 'neo' | 'handDrawn';

// Add to shared state:
look: DiagramLook;
setLook: (look: DiagramLook) => void;
```

Default value: `'default'`

#### 1.2 Add Look Selector UI

Create a segmented control / radio group with three options. Place it:
- In the Apply tab: immediately below the palette strip, as a compact inline control
- In the Compose tab: as a full-width section with descriptions

```
Look selector labels and descriptions:
┌─────────┬──────────┬────────────┐
│ Classic │   Neo    │   Sketch   │
└─────────┴──────────┴────────────┘

Classic  = "Clean flat rendering. Maximum compatibility."
Neo      = "Modern with drop shadows and refined borders. Requires Mermaid v11.14+."
Sketch   = "Hand-drawn rough style. Informal, whiteboard aesthetic."
```

Styling:
- Selected option has a filled background using the current theme's accent color
- Unselected options have outline/ghost styling
- On mobile, the three options should be equally sized buttons in a horizontal row

#### 1.3 Update Theme Engine

In `themeEngine.ts` (or wherever the init directive is generated), include the `look` property in the output:

**Before (current):**
```json
%%{init: { "theme": "base", "themeVariables": { ... } }}%%
```

**After (updated):**
```json
%%{init: { "theme": "base", "look": "neo", "themeVariables": { ... } }}%%
```

When `look` is `'default'`, omit the `look` property entirely (it's the default and saves directive space).

When `look` is `'neo'` or `'handDrawn'`, include `"look": "{value}"` in the init directive.

#### 1.4 Update ALL Exports

Every export path (Styled Code, Markdown Bootstrap, Prompt Scaffold, Theme JSON) must include the selected look:
- **Styled Code:** Include `look` in the init directive
- **Markdown Bootstrap:** Add a "Look" row to the bootstrap header
- **Prompt Scaffold:** Instruct the LLM to use the specified look
- **Theme JSON:** Include `"look": "neo"` in the exported JSON

#### 1.5 Update Preview

The Mermaid preview renderer must pass the `look` config when calling `mermaid.render()` or `mermaid.initialize()`:

```ts
mermaid.initialize({
  theme: 'base',
  look: selectedLook, // 'default' | 'neo' | 'handDrawn'
  themeVariables: { ... }
});
```

Verify that toggling between Classic/Neo/Sketch re-renders the preview with the appropriate visual treatment.

## SECTION 2: TYPOGRAPHIC HIERARCHY SYSTEM (New Feature)

### Concept

Diagrams have a typographic hierarchy analogous to document headings. In a well-structured document:
- The document title is H1 (largest, most prominent)
- Section headings are H2
- Subsection headings are H3
- Body text is smaller than any heading

In a Mermaid diagram:
- The **diagram title** is the H1 equivalent
- **Node/shape labels** are the body text equivalent
- **Subgraph titles** are the H2 equivalent (they label sections)
- **Nested subgraph titles** should never be larger than their parent subgraph titles (H3 or smaller)
- **Edge labels** are annotation text (smallest)

The Theme Builder should let users control font family, font size, and font color for each of these typographic tiers independently.

### 2.1 Define Typography Tier Data Model

```tsx
interface TypographyTier {
  fontFamily: string;
  fontSize: number;       // in px
  fontColor: string;      // hex color
}

interface TypographySettings {
  diagramTitle: TypographyTier;        // H1: largest, diagram-level title
  subgraphTitle: TypographyTier;       // H2: subgraph/section labels
  nestedSubgraphTitle: TypographyTier; // H3: nested subgraph labels
  nodeLabel: TypographyTier;           // Body: shape/node text
  edgeLabel: TypographyTier;           // Annotation: edge/link labels
}
```

Default values (sensible starting point):

```tsx
const DEFAULT_TYPOGRAPHY: TypographySettings = {
  diagramTitle: {
    fontFamily: 'Trebuchet MS',
    fontSize: 20,
    fontColor: '#1C3A34',
  },
  subgraphTitle: {
    fontFamily: 'Trebuchet MS',
    fontSize: 16,
    fontColor: '#2D6F7E',
  },
  nestedSubgraphTitle: {
    fontFamily: 'Trebuchet MS',
    fontSize: 14,
    fontColor: '#2D6F7E',
  },
  nodeLabel: {
    fontFamily: 'Trebuchet MS',
    fontSize: 14,
    fontColor: '#111827',
  },
  edgeLabel: {
    fontFamily: 'Trebuchet MS',
    fontSize: 12,
    fontColor: '#666666',
  },
};
```

### 2.2 Add Typography to Shared State

```tsx
// Add to app state
typography: TypographySettings;
setTypography: (typography: TypographySettings) => void;
setTypographyTier: (tier: keyof TypographySettings, values: Partial<TypographyTier>) => void;
```

### 2.3 Hierarchy Enforcement Rule

**Critical constraint:** A nested subgraph title font size must NEVER exceed its parent subgraph title font size. When the user changes a parent tier's font size, automatically cap any child tier that would violate the hierarchy.

Enforcement logic:
```tsx
function enforceHierarchy(typography: TypographySettings): TypographySettings {
  const result = { ...typography };

  // Diagram title is always the largest (no enforcement needed, user controls it)

  // Subgraph title must not exceed diagram title
  if (result.subgraphTitle.fontSize > result.diagramTitle.fontSize) {
    result.subgraphTitle.fontSize = result.diagramTitle.fontSize;
  }

  // Nested subgraph title must not exceed subgraph title
  if (result.nestedSubgraphTitle.fontSize > result.subgraphTitle.fontSize) {
    result.nestedSubgraphTitle.fontSize = result.subgraphTitle.fontSize;
  }

  // Node label: no strict hierarchy enforcement against subgraphs,
  // but should typically be <= subgraph title. Show a warning if larger.

  // Edge labels should be smallest. Show a warning if larger than node labels.

  return result;
}
```

When a violation occurs, show a brief inline warning: "Adjusted: nested subgraph title can't exceed subgraph title."

### 2.5 Map Typography to Mermaid Config

Mermaid's themeVariables support some of these directly. Others require classDef or CSS injection.

**What maps to themeVariables directly:**

```json
{
  "fontFamily": "Trebuchet MS",
  "fontSize": "14px",
  "titleColor": "#1C3A34"
}
```

The `fontFamily` and `fontSize` in themeVariables apply globally. To differentiate tiers, the theme engine must generate **classDef** blocks and **CSS** overrides:

**Generated output for typography (flowchart example):**

```mermaid
%%{init: {
  "theme": "base",
  "themeVariables": {
    "fontFamily": "Trebuchet MS",
    "fontSize": "14px",
    "titleColor": "#1C3A34"
  }
}}%%
flowchart TD
  %% Typography classDefs (generated by Mermaid Theme Builder)
  classDef nodeStyle font-family:Trebuchet MS,font-size:14px,color:#111827
  classDef edgeLabelStyle font-family:Trebuchet MS,font-size:12px,color:#666666

  subgraph Section1["Section Title"]
    A["Node A"] --> B["Node B"]
  end

  %% Note: Subgraph title styling requires CSS injection or
  %% init-level cluster config. See implementation notes below.
```

**For subgraph title styling**, Mermaid supports these themeVariables:
- `clusterBkg` (already in the color system)
- Subgraph labels inherit the global fontFamily/fontSize

To differentiate subgraph title sizes from node label sizes, the theme engine should:

1. Set the global `fontSize` to the **node label** size (since nodes are the most common element)
2. Set `titleColor` to the diagram title color
3. Generate a `<style>` block or CSS injection for subgraph-specific overrides:

```css
/* Injected by Mermaid Theme Builder */
.cluster-label {
  font-family: 'Trebuchet MS' !important;
  font-size: 16px !important;
  fill: #2D6F7E !important;
}
.edgeLabel {
  font-family: 'Trebuchet MS' !important;
  font-size: 12px !important;
  fill: #666666 !important;
}
```

Mermaid supports inline `<style>` blocks in diagrams for CSS customization. The theme engine should append these when typography tiers differ from the global defaults.

**Important: NOT all diagram types support CSS injection.** The theme engine should:
- For flowcharts: use CSS injection (most flexible)
- For sequence diagrams: use themeVariables (`actorTextColor`, `signalTextColor`, `noteTextColor`, etc.)
- For other types: use the closest available themeVariable mapping
- Always include a comment in exported code noting which typography tiers are approximate

**After (correct):**


```markdown
# Contributing to Mermaid Theme Builder

Thank you for your interest in contributing.

## Quick Start

1. Fork the repo
2. Clone your fork
3. `pnpm install`
4. `pnpm --filter @workspace/mermaid-theme-builder dev`
5. Make your changes
6. `pnpm --filter @workspace/mermaid-theme-builder build` (verify it compiles)
7. Submit a PR

## Brand Rules (Non-Negotiable)

This is a personal OverKill Hill P3 project. See AGENTS.md for brand separation rules.

- Do NOT add preset themes named after third-party companies
- Do NOT reference BFS, Builders FirstSource, or any employer
- Do NOT use these hex values: #00205B, #003087, #002F86, #B3C1DB, #D6E5F9, #D0D0CE, #C8102E
- Use generic palette names (Ocean Depth, Forest Sage, etc.) or personal brand names only

## Code Style

- TypeScript strict mode
- Functional React components with hooks
- Tailwind CSS for styling
- No server-side code (100% client-side)

## What We Accept

- Bug fixes with repro steps
- New palette presets (generic names only)
- New example diagrams
- Accessibility improvements
- Documentation improvements

## What We Don't Accept

- Server-side features (auth, storage, API calls)
- Third-party brand presets
- AI API integrations (V1 is static)
- Features that break GitHub Pages compatibility

MIT. All contributions are under the same license.
```
